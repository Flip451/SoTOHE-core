# Adr2pr Workflow SSoT

> Provider-agnostic workflow SSoT for the `adr2pr` track workflow. Both the Claude adapter
> (`.claude/commands/track/adr2pr.md`) and the Codex skill adapter
> (`.agents/skills/track-adr2pr/SKILL.md`) reference this file. Provider-specific invocation
> framing lives in those adapters; the full workflow contract lives here.

## Mission

Drive a prepared ADR all the way to a reviewed PR — autonomously sequencing init → review →
commit → planning phases → review → commit → full-cycle → pr-review — without merging. This is
a thin orchestrator: it sequences existing sub-workflows and adds only the adr2pr-specific
ordering and constraints. It does NOT duplicate sub-workflow internals; each step delegates to
its canonical sub-workflow, which owns its own gates / parallelism / writer ownership /
back-and-forth escalation.

**Precondition**: a pre-track ADR must already exist under `knowledge/adr/`. If no relevant ADR
exists, stop and ask the user to author one before running this workflow.

Sub-workflows used (in execution order):

- `.harness/workflows/track/init.md`
- `.harness/workflows/track/review.md`
- `.harness/workflows/track/commit.md`
- `.harness/workflows/track/spec-design.md`
- `.harness/workflows/track/type-design.md`
- `.harness/workflows/track/impl-plan.md`
- `.harness/workflows/track/full-cycle.md`
- `.harness/workflows/track/pr-review.md`

## Inputs

- **ADR** — a pre-track ADR under `knowledge/adr/`. If none exists, stop and ask the user to
  author one.
- **Track initialization state** — Step 1 is conditional: if a `track/<id>` branch already
  exists with `metadata.json`, skip `init` and start at Step 2.
- **Autonomy level** — this workflow is fully autonomous (Constraint 2). All commit messages are
  generated by the workflow; no candidate presentation or user confirmation is required for
  any step. The workflow stops only on irrecoverable hard failures.

## Sequence

**Step 0 (required before execution): Build an execution plan**

Read every sub-workflow definition referenced in this workflow and extract their gates /
decision points / parallelism rules into a concrete execution plan. Treat them as a state
machine to execute. This is the first action; no execution step may begin until the plan
is complete.

**Step 1: init workflow (conditional)**

If the track is already initialized (on a `track/<id>` branch with `metadata.json`), skip
this step and start at Step 2. Otherwise, invoke the `init` workflow
(`.harness/workflows/track/init.md`) to create the track directory, write `metadata.json`,
and materialize the branch. On ERROR, stop and report.

**Step 2: review workflow — ADR baseline**

Invoke the `review` workflow (`.harness/workflows/track/review.md`) to review the ADR
baseline. At this stage `spec.md` does not exist yet (created in Step 5). The `review`
workflow scopes to what exists: `plan.md`, `metadata.json`, and the ADR files in the working
tree. The absence of `spec.md` does not block the review.

**Step 3: Stage**

Run `cargo make add-all` after the final review round (per the canonical staging-order rule in
the `commit` workflow: review must complete before staging).

**Step 4: commit workflow — ADR + metadata commit**

Invoke the `commit` workflow (`.harness/workflows/track/commit.md`). Generate the commit
message per Constraint 2 (no candidate presentation). This is the first commit on the track
branch.

**Step 5: Planning phases (Phase 1-3, sequential)**

Invoke the `spec-design`, `type-design`, and `impl-plan` workflows directly in sequence.
Each workflow is single-shot (invoke once, receive signal / gate result). Back-and-forth
escalation after 🔴 / ERROR is this orchestrator's responsibility; apply the loop rules
defined in the `plan` workflow SSoT's "Phase 1 loop", "Phase 2 loop", and "Phase 3 loop"
sections (see `.harness/workflows/track/plan.md`; do not re-state them here).

Do NOT invoke the `plan` workflow: after Step 1, the orchestrator is on a `track/<id>` branch,
so the `plan` workflow's Phase 0 (`init`, which requires the configured base branch) would fail;
and the `plan`
workflow's Termination section can pause for a user decision when ADR working-tree diffs are
present (violates Constraint 2). Plan artifacts are staged at Step 6 and committed at Step 7.

Signal resolution rule for 🟡 (yellow): the `plan` workflow allows 🟡 to advance with a
warning, but this `adr2pr` workflow requires 🟡 to be resolved before Step 9 (`full-cycle`
begins). Any remaining 🟡 at Step 10 (`pr-review`) will surface as a merge blocker. If Phase 1
or Phase 2 returns 🟡 after all reds are cleared, re-invoke the relevant writer to address
the yellow before proceeding to the next step.

**Step 6: review workflow — plan artifacts**

Invoke the `review` workflow (`.harness/workflows/track/review.md`) to review the plan
artifacts (`spec.json`, type catalogues, `impl-plan.json`).

**Step 7: Stage**

Run `cargo make add-all` after the final review round.

**Step 8: commit workflow — plan artifacts commit**

Invoke the `commit` workflow. Generate the commit message per Constraint 2. This is the second
commit on the track branch.

**Step 9: full-cycle workflow**

Invoke the `full-cycle` workflow (`.harness/workflows/track/full-cycle.md`) for
feature-batch implement → DRY check → review → commit. See that workflow's SSoT for the
full internal state machine.

**Step 10: pr-review workflow**

Invoke the `pr-review` workflow (`.harness/workflows/track/pr-review.md`) for the final
PR-based review (whole track branch vs the configured merge target). The workflow stops at
reviewed PR. No merge
is performed.

## Gates

| Step | Gate | Verdict |
|------|------|---------|
| 1 | `init` OK or pre-existing track | proceed / ERROR |
| 2 | `review` check-approved exit 0 | proceed / blocked |
| 4 | `commit` CI + git commit OK | proceed / ERROR |
| 5 | Phase 1-3 gates (per `plan` SSoT loop rules) | proceed / escalate / stop |
| 6 | `review` check-approved exit 0 (plan artifacts) | proceed / blocked |
| 8 | `commit` CI + git commit OK | proceed / ERROR |
| 9 | `full-cycle` completes (all batches committed) | proceed / stop |
| 10 | `pr-review` reaches explicit zero-findings terminal state | PASS / loop |

All gates are binary; no step begins until the previous step's gate passes.

## Failure / recovery

- **No ADR**: stop and ask the user to author one before running this workflow.
- **irrecoverable step failure**: stop and report. Autonomy (Constraint 2) covers
  decision-making, not ignoring hard failures.
- **Phase 1-3 🔴 / ERROR after `max_retry` overflow**: stop and present options (continue
  with warnings, abort, or manual edit). The user decides.
- **DFP `blocked`** (in `full-cycle`): surface violation pairs and halt. Do not proceed to
  review or commit.
- **PR review loop actionable findings**: fix locally, commit, re-run `pr-review`. Repeat
  until explicit zero findings. The loop MUST continue until the terminal condition is met.

## Constraints

1. **No merge**: the workflow stops at Step 10 (`pr-review`). Never invoke any merge workflow.
   Leave the PR open for the user to merge.
2. **Fully autonomous**: never pause for user confirmation at any step; run to the end. All
   commit messages are generated by the workflow (no candidate presentation).
3. **Batch-first execution**: `full-cycle` implements the full feature batch before review and
   commit by default. Per-task commit split is triggered only when a layer's cumulative diff
   is about to exceed its per-scope ceiling.
4. **Signals must be resolved**: resolve every 🔴 at the phase where it surfaces. For 🟡:
   resolve all yellows before Step 9 (`full-cycle`) begins; any remaining 🟡 at Step 10 will
   surface as a merge blocker for the user.

## Outputs

- Track branch with commits: ADR + metadata commit, plan artifacts commit, feature batch
  commit(s), optional lifecycle tail commit
- Pull request (created or reused; not merged)
- PR URL and terminal `pr-review` state
- Per-step gate verdicts and commits produced
- Any per-scope ceiling batch split decisions made during `full-cycle`
- Confirmation that all 🔴/🟡 signals are resolved before Step 9
