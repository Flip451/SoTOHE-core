## Type Design

### Decision rationale per type

**`ParamDeclaration` — struct, not enum.**
A parameter always has both a name and a type string. There are no valid states where one is present and the other is absent. No state transition exists. The coding-principles decision table says: "有限の値の集合（遷移なし）→ enum-first" does not apply because `ParamDeclaration` is not a finite set — it is a record with two always-present fields. A struct is correct here.

**`MethodDeclaration` — struct, not enum.**
A method declaration has five fields that are all simultaneously meaningful: name, receiver, params, returns, is_async. Unlike `MemberDeclaration` (see below), no field is conditionally absent based on a discriminant. There is no state transition. A struct with `Option<String>` for `receiver` (nullable in the JSON, `"&self"` / `"&mut self"` / `"self"` or absent for associated functions) is the right shape. The `Option<String>` for `receiver` represents a genuine optional value (not a discriminated state), so it does not violate enum-first: "struct + `Option<T>` で「この状態では None」→ enum を検討" — here, `receiver = None` is NOT tied to a special state of the method; any async/sync method can be an associated function. The `Option` is independent of other fields.

**`MemberDeclaration` — enum (enum-first).**
The decision between `struct MemberDeclaration { name: String, ty: Option<String> }` and `enum MemberDeclaration { Variant(String), Field { name: String, ty: String } }` must follow the coding-principles table. The key question: does `ty = None` represent a genuine absent value, or does it represent a specific state (enum variant)? In Rust, enum variants never have declared types in TDDD (we only track variant names); struct fields always have a type. This is a 1:1 correspondence between the `None` state and the semantic "variant" concept. The coding-principles rule says: "struct + Option\<T\> で「この状態では None」→ enum を検討（Option の None が特定状態と 1:1 対応なら enum が適切）". Therefore `enum MemberDeclaration` is the correct choice.

**`FunctionInfo` extensions — structured fields added, `signature` removed (C1 applied).**

///

/// making the two semantically distinct states structurally unambiguous.
#[derive(Debug, Clone, PartialEq, Eq, PartialOrd, Ord, Hash, serde::Serialize, serde::Deserialize)]
#[serde(untagged)]

    Variant(String),
    /// A struct field: a name and its type string.
    Field { name: String, ty: String },
}

    /// Returns the member name regardless of variant.
    #[must_use]
    pub fn name(&self) -> &str {
        match self {
            Self::Variant(n) => n,
            Self::Field { name, .. } => name,
        }
    }

    /// Returns the field type, if this is a struct field.
    #[must_use]
    pub fn ty(&self) -> Option<&str> {
        match self {
            Self::Variant(_) => None,
            Self::Field { ty, .. } => Some(ty),
        }
    }

### Q1 — Catalogue file split structure (DM-06)

Recommended split:
- `libs/domain/src/tddd/catalogue.rs` (~300 lines): types only — `TypeAction`, `TypestateTransitions`, `TypeDefinitionKind`, `TypeCatalogueEntry`, `TypeCatalogueDocument`, `MethodDeclaration`, `ParamDeclaration`, `MemberDeclaration`, `TypeSignal`
- `libs/domain/src/tddd/signals.rs` (~350 lines): `evaluate_type_signals`, `evaluate_single`, per-kind evaluators (`evaluate_typestate`, `evaluate_enum`, `evaluate_value_object`, `evaluate_error_type`, `evaluate_trait_port`), `evaluate_delete`, helper constructors (`red`, `yellow`, `blue`), `undeclared_to_signals`
- `libs/domain/src/tddd/consistency.rs` (~320 lines): `ActionContradiction`, `ActionContradictionKind`, `ConsistencyReport`, `check_consistency`, `check_type_signals`

Rationale: The rename touches virtually every line, so a concurrent structural split has zero incremental cost. The three-way split follows the single-responsibility principle: type declarations (stable contract), signal computation (algorithmic logic), consistency orchestration (cross-cutting). The split also keeps each module under the 400-line warn threshold from `architecture-rules.json`. `check_type_signals` (formerly `check_domain_types_signals`) is placed in `consistency.rs` because it is the signal gate function that depends on both `TypeSignal` (defined in `catalogue.rs`) and `ConsistencyReport` (defined in `consistency.rs`).

### Q2 — `MethodDeclaration` vs `signature: String`

> **[SUPERSEDED by ADR C1 — 2026-04-12]** The answer below was the initial planner proposal.
> After user review, investigation confirmed that `.signature()` has **zero callers** in
> `libs/` or `apps/` (verified by grep). The BRIDGE-01 concern did not apply. The ADR C1
> decision overrides this Q2 answer: **delete** `FunctionInfo::signature: String` and the
> `pub fn signature()` accessor entirely. The authoritative guidance is in

> `track/items/tddd-01-multilayer-2026-04-12/metadata.json` S2. This note is preserved as

Keep `signature: String` as an independent field on `FunctionInfo`. Rationale: `SchemaExport` is consumed by BRIDGE-01 outside TDDD; the `signature` string is the primary human-readable output in the `export-schema` feature. Removing or recomputing it would break BRIDGE-01 callers. On `MethodDeclaration` (catalogue/TypeGraph/baseline), there is no `signature` string — the structured fields are the source of truth and a `signature_string(&self) -> String` method can reconstruct it on demand for display purposes. This avoids dual-maintenance of `signature` on the catalogue type while preserving it where it is genuinely needed (`FunctionInfo` for BRIDGE-01).

### Q3 — `MemberDeclaration` type shape

Use `enum MemberDeclaration { Variant(String), Field { name: String, ty: String } }` (enum-first per coding principles). Rationale as stated in Type Design section: `ty = None` for enum variants is a 1:1 correspondence with the "variant" semantic state, making an enum structurally correct. The untagged serde representation makes the JSON compact: `"Variant"` for enum variants, `{"name": "field_name", "ty": "TypeName"}` for struct fields. Existing `TypeInfo::members: Vec<String>` was used for both variant names and field names uniformly, which meant type info for struct fields was always absent. The enum split makes this explicit at the type level.

The migration note: existing signal evaluators that compare `members()` now compare `MemberDeclaration` values. For `evaluate_enum` and `evaluate_error_type` (which only care about variant names), callers call `m.name()`. For struct field matching (currently not implemented in any evaluator), the `Field` variant is available.

### Q4 — Typestate `outgoing` derivation from structured methods

Use approach (b): introduce a helper `fn first_return_type_name(s: &str) -> &str` that returns the outermost type name from a generic type expression. Implementation: scan the string left-to-right for the first `<` character; the outermost type name is the substring `[0, first_angle_bracket_or_end)`. Edge cases:

- `"User"` → `"User"`
- `"Result<Option<User>, DomainError>"` → `"Result"` (for typestate target extraction)
- But wait: the typestate detection needs `User`, not `Result`

The ADR says method return types are used to detect typestate targets. The current code uses `return_type_names` which already does the stripping (via `extract_return_type_names` in `RustdocSchemaExporter`). With structured methods, `MethodDeclaration::returns` holds the full string `"Result<Option<User>, DomainError>"`.

Decision: keep the existing `extract_return_type_names` logic on `FunctionInfo` for populating a parallel `return_type_names` field, and use that field (not the structured `returns` string) for typestate `outgoing` derivation in `build_type_graph`. This is consistent with current behavior and avoids introducing a new parsing function. The `outgoing` derivation in `TypeNode::new` continues to use `return_type_names` filtered by `typestate_names`. The structured `returns` field on `MethodDeclaration` is used for L1 forward check (exact string match against the catalogue declaration), not for typestate graph construction.

Concretely: `build_type_graph` still collects `method_return_types: HashSet<String>` as a local variable from `FunctionInfo::return_type_names()`, uses it to compute `outgoing`, and does NOT store it on `TypeNode` (removed field). The `outgoing` field on `TypeNode` is populated from this local computation. `TypeNode::methods` stores the full `Vec<MethodDeclaration>`.

## Edge Cases

### From briefing (confirmed)

1. **All `tddd.enabled: false`** — `verify_from_spec_json` and `check_strict_merge_gate` loop over zero layers; Stage 2 is entirely skipped. Behavior identical to pre-multilayer.

2. **Partial enable** — Only enabled-layer catalogues are read. A layer with `enabled: false` never has its catalogue file checked, even if the file happens to exist in the track directory.

3. **Layer enabled but no catalogue in track** — `NotFound` → skip (TDDD opt-in per layer). This is the same semantics as the current single-layer NotFound behavior. It is NOT an error. A misconfiguration where the catalogue_file is wrong (e.g., typo in `architecture-rules.json`) would silently skip; the developer discovers this when they expect signals but see "skipped". Recommend a CLI warning: `[WARN] layer 'usecase' has tddd.enabled=true but no catalogue found at '<path>' — run 'sotp track type-signals --layer usecase' to create it.`

4. **Parallel baseline-capture** — The CLI's default mode (all layers) writes each baseline atomically (via `atomic_write_file`). Two concurrent invocations may race on the same baseline file. The `--force` flag is needed to regenerate; without it, the idempotent check may cause the second invocation to skip. This is acceptable; the Makefile should serialize the invocations.

5. **Catalogue/baseline naming drift** — Enforced by rule: the baseline file name is derived deterministically from `catalogue_file` as `<stem>-baseline.json`. The codec accepts only the versioned schema; a filename mismatch (e.g., `usecase-types.json` paired with `domain-types-baseline.json`) would cause a file-not-found error, not a silent mismatch.

6. **Symlink guard coverage** — Must apply individually to every layer's catalogue AND baseline path. The existing `reject_symlinks_below` is called once per file per layer in both `verify_from_spec_json` and `baseline_capture`.

7. **Strict gate U3 multilayer** — ANY Yellow in ANY enabled layer blocks the whole gate. AND-merge of `VerifyOutcome` findings means the first `has_errors()` short-circuits per the current `check_strict_merge_gate` pattern. In multilayer, we do NOT short-circuit after the first failing layer — we collect all layer findings and return the union, so the developer sees all issues at once. The gate still blocks.

8. **Delete+Add pair + MethodDeclaration** — `Delete` action checks presence/absence of the type in the graph partition, not its signature. With `MethodDeclaration` in `TraitPort`, the `evaluate_delete` function checks `profile.get_trait(name).is_some()` — this is a name-only lookup, unchanged by the signature extension. The forward check for the paired `Add` TraitPort entry uses the new L1 signature check. No regression.

9. **Typestate outgoing derivation** — Uses `FunctionInfo::return_type_names` (the existing extracted set) for outgoing computation. `Result<Foo, Bar>` → `return_type_names = ["Foo"]` via the existing `extract_return_type_names` logic (strips outer Result/Option/etc.). This is consistent with current behavior.

10. **ActionContradiction for `Reference`** — `Reference` action requires the forward signal to be Blue. With MethodDeclaration, Blue now requires strict signature match. A `Reference` entry for a `TraitPort` that was formerly Blue (name-only match) may now be Yellow if the actual signature differs from the declaration. This is a deliberate behavior improvement, not a regression. The migration note: existing tracks using `Reference` for TraitPort entries should re-run `type-signals` after the update; if Yellow appears, they must update the `expected_methods` declarations to match the actual code. Include this in the breaking change notes.

### Additional edge cases (not in briefing)

**EC-A: Empty `targets` array in `schema_export`** — `"targets": []` in `architecture-rules.json` is syntactically valid but semantically degenerate (no crates to export). The CLI should treat this as a configuration error: `"[ERROR] layer 'domain' has tddd.enabled=true but schema_export.targets is empty — add at least one crate name."` Fail fast before invoking rustdoc.

**EC-B: `catalogue_file` collision across layers** — Two layers with the same `catalogue_file` name (e.g., both pointing to `types.json`) would overwrite each other's catalogue in the track directory. The CLI `type-signals` command should validate uniqueness of `catalogue_file` values across enabled layers and fail with an error if duplicates are detected.

**EC-C: `MethodDeclaration` ordering in forward check** — The L1 forward check compares `params` in order (step 3 in ADR D2). If the rustdoc JSON returns parameters in a different order than declared (unlikely but possible with proc-macro-generated traits), the forward check produces a false Yellow. Mitigation: document that parameter order must match the declaration order. Do NOT sort params — order is part of the signature contract.

**EC-D: `is_async` and `async-trait` desugaring** — When traits use `async-trait` proc macro, the generated methods are not `async fn` at the trait level; they return `Pin<Box<dyn Future>>`. Rustdoc JSON will report `is_async = false` for such methods. If a developer declares `"async": true` in the catalogue, the forward check produces Yellow. This is a known limitation of L1 (noted in ADR Consequences). The migration note: use `"async": false` for trait methods implemented via `async-trait` until native async fn in traits are used throughout.

**EC-E: `MemberDeclaration::Field` type normalization** — Struct field types in rustdoc JSON may include full module paths (`domain::user::UserId`). The `last_segment` normalization must be applied at extraction time (in `RustdocSchemaExporter`). If a developer writes `{ "Field": { "name": "id", "ty": "domain::user::UserId" } }` in a catalogue, the forward check will fail because the extracted value is `UserId`. Enforce the normalization rule in the codec validation: reject any `ty` string containing `::` with a helpful error message directing the user to use the last-segment short name.

**EC-F: `TraitNode.methods` ordering for `structurally_equal`** — `TraitBaselineEntry.methods: Vec<MethodDeclaration>` comparison must be order-insensitive (methods can be declared in any order in the trait). Sort by method name before comparison. The `structurally_equal` function should sort both slices before comparing. `MethodDeclaration` implements `Ord` (derived `PartialOrd`/`Ord` based on name as primary key). Similarly for `TypeBaselineEntry.methods`.

**EC-G: Schema version mismatch on `read_type_catalogue` in merge gate** — The merge gate reads `<catalogue_file>` via `git show`. If the file was written by a pre-TDDD-01 CLI (schema_version: 1 with `domain_types` key and string `expected_methods`), the decode will fail. This is a `FetchError` scenario → BLOCKED. The error message should be specific: `"domain-types.json: unsupported schema_version: expected 2, got 1 — re-run 'sotp track type-signals' after upgrading TDDD-01."` This prevents silent pass-through of stale files.

### Block 3: `MethodDeclaration` Rust struct (domain layer)

```rust
// libs/domain/src/tddd/catalogue.rs

/// A single parameter in a method signature (L1 resolution).
///
/// The `name` is the parameter binding identifier; `ty` is the type string
/// with last-segment short names and generics preserved verbatim.
#[derive(Debug, Clone, PartialEq, Eq, PartialOrd, Ord, Hash)]
#[derive(serde::Serialize, serde::Deserialize)]
pub struct ParamDeclaration {
    pub name: String,
    pub ty: String,
}

/// A member of a composite type: either an enum variant or a struct field.
///
/// Enum-first design: the two states carry structurally distinct data —
/// a variant has only a name; a field has a name and a type string.
/// Using `Option<String>` for `ty` would allow the illegal state
/// `ty = None` on a struct field, which this enum prevents.
#[derive(Debug, Clone, PartialEq, Eq, PartialOrd, Ord, Hash)]
#[derive(serde::Serialize, serde::Deserialize)]
#[serde(untagged)]
pub enum MemberDeclaration {
    /// An enum variant (name only; payload types are not tracked at L1).
    Variant(String),
    /// A struct field with its type string.
    Field { name: String, ty: String },
}

impl MemberDeclaration {
    /// Returns the member name regardless of kind.
    #[must_use]
    pub fn name(&self) -> &str {
        match self {
            Self::Variant(n) => n,
            Self::Field { name, .. } => name,
        }
    }

    /// Returns the field type, or `None` for enum variants.
    #[must_use]
    pub fn ty(&self) -> Option<&str> {
        match self {
            Self::Variant(_) => None,
            Self::Field { ty, .. } => Some(ty),
        }
    }
}

/// A structured method signature at L1 resolution.
///
/// Shared across three contexts:
/// - Catalogue declaration: `TypeDefinitionKind::TraitPort { expected_methods }`
/// - TypeGraph: `TypeNode::methods` / `TraitNode::methods`
/// - Baseline: `TypeBaselineEntry::methods` / `TraitBaselineEntry::methods`
///
/// Type strings use last-segment short names; generics are preserved verbatim
/// (e.g., `"Result<Option<User>, DomainError>"`, not `"Result"`).
/// Module paths must NOT appear (no `::`) — reject at codec decode time.
///
/// # Errors
///
/// No construction errors; all fields are unchecked strings.
/// Codec validation rejects strings containing `::` in `ty` / `returns` fields.
#[derive(Debug, Clone, PartialEq, Eq)]
#[derive(serde::Serialize, serde::Deserialize)]
pub struct MethodDeclaration {
    /// Method name (snake_case Rust identifier).
    pub name: String,
    /// Self-receiver form: `"&self"`, `"&mut self"`, `"self"`, or `None`
    /// for associated functions.
    pub receiver: Option<String>,
    /// Ordered parameter list (empty for methods with no non-self parameters).
    pub params: Vec<ParamDeclaration>,
    /// Return type string (`"()"` when the return type is the unit type).
    pub returns: String,
    /// Whether the method is declared `async fn`.
    #[serde(rename = "async")]
    pub is_async: bool,
}
```

// libs/domain/src/schema.rs

use crate::tddd::catalogue::{MemberDeclaration, MethodDeclaration};

/// A public type in the crate (post Phase 1 extension).
#[derive(Debug, Clone)]
pub struct TypeNode {
    kind: TypeKind,
    /// Struct fields or enum variants with type info.
    /// Sorted for deterministic comparison.
    members: Vec<MemberDeclaration>,
    /// Full method signatures from inherent (non-trait) impl blocks.
    /// `outgoing` is derived from these at construction.
    methods: Vec<MethodDeclaration>,
    /// Outgoing typestate transitions: type names returned by `methods`
    /// that are registered in `typestate_names` at build time.
    outgoing: HashSet<String>,
    /// Module path for disambiguation (e.g., `"domain::review"`).
    module_path: Option<String>,
}

impl TypeNode {
    /// Creates a new `TypeNode`.
    ///
    /// `outgoing` is supplied pre-computed by `build_type_graph` (derived from
    /// `FunctionInfo::return_type_names ∩ typestate_names`).
    #[must_use]
    pub fn new(
        kind: TypeKind,
        members: Vec<MemberDeclaration>,
        methods: Vec<MethodDeclaration>,
        outgoing: HashSet<String>,
    ) -> Self {
        Self { kind, members, methods, outgoing, module_path: None }
    }

    /// Sets the module path.
    pub fn set_module_path(&mut self, path: String) {
        self.module_path = Some(path);
    }

    /// Returns the kind of this type.
    #[must_use]
    pub fn kind(&self) -> &TypeKind { &self.kind }

    /// Returns the member declarations (fields or variants).
    #[must_use]
    pub fn members(&self) -> &[MemberDeclaration] { &self.members }

    /// Returns the inherent impl method declarations.
    #[must_use]
    pub fn methods(&self) -> &[MethodDeclaration] { &self.methods }

    /// Returns the outgoing typestate transitions.
    #[must_use]
    pub fn outgoing(&self) -> &HashSet<String> { &self.outgoing }

    /// Returns the module path, if known.
    #[must_use]
    pub fn module_path(&self) -> Option<&str> { self.module_path.as_deref() }
}

/// A public trait in the crate (post Phase 1 extension).
#[derive(Debug, Clone)]
pub struct TraitNode {
    /// Full method signatures defined by this trait.
    /// Sorted by name for deterministic comparison.
    methods: Vec<MethodDeclaration>,
}

impl TraitNode {
    /// Creates a new `TraitNode`.
    #[must_use]
    pub fn new(methods: Vec<MethodDeclaration>) -> Self {
        Self { methods }
    }

    /// Returns the method declarations of this trait.
    #[must_use]
    pub fn methods(&self) -> &[MethodDeclaration] { &self.methods }
}

**Most controversial type design choice:**
`MemberDeclaration` as an enum rather than `struct MemberDeclaration { name: String, ty: Option<String> }`. The controversy: the `untagged` serde representation makes `Variant("Name")` serialize as a plain JSON string, while `Field { name, ty }` serializes as an object. This means the JSON array can contain mixed types (`["Variant", {"name": "field", "ty": "String"}]`), which is unusual and may surprise developers reading the baseline JSON manually. The justification is that coding-principles explicitly require enum-first when `Option = None` maps 1:1 to a semantic state, and enum variants vs struct fields are precisely such a state distinction. The baseline JSON verbosity is acceptable given that it is machine-generated.

**Additional edge case not in the briefing:**
EC-G (schema version mismatch on `read_type_catalogue` in the merge gate): if a track was written with the old CLI before TDDD-01 was deployed, the merge gate will encounter a v1 catalogue file and produce a `FetchError` rather than a silent pass. This is the correct fail-closed behavior, but the error message must be specific enough for the developer to understand why the gate is blocking a previously-passing branch.
