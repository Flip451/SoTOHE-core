In `rustdoc-types` 0.57, the representation of a trait implementation (`impl Trait for Struct`) is encapsulated in the `Item` enum's `Impl` variant. The structure is designed to handle both inherent implementations and trait implementations, including complex cases like generic, blanket, and synthetic (auto-trait) implementations.

### 1. Representation of `impl Trait for Struct` in `Item::Impl`

A trait implementation is an `Item` where `inner` is `ItemEnum::Impl(Impl)`. The `Impl` struct contains the following key fields (Gemini output, mostly verified for version 0.57 — see Addendum for `is_negative` caveat):

- **`trait_: Option<Path>`**: Contains the trait being implemented. If `None`, it's an inherent impl (`impl Struct {}`).
- **`for_: Type`**: The type that the implementation is for (the "receiver").
- **`items: Vec<Id>`**: Identifiers for the items defined within this block (e.g., methods, constants).
- **`generics: Generics`**: Generic parameters and where-clauses for the `impl` block itself.
- **`is_negative: bool`**: `true` for negative implementations like `impl !Send for MyType`. (**Gemini output only — not confirmed against codebase; see Addendum.**)
- **`is_synthetic: bool`**: `true` for auto-trait implementations (e.g., `Send`, `Sync`, `Sized`) generated by the compiler.
- **`blanket_impl: Option<Type>`**: If this is a blanket implementation (e.g., `impl<T> Trait for T`), this field contains the type it applies to.

### 2. Extracting the Trait Name

In **rustdoc-types 0.57**, the `Path` struct uses the field name **`path`** (renamed from `name` in older versions) to store the string representation.

- **Simple Name**: `Impl.trait_.as_ref().unwrap().path` provides the name as written in the source (e.g., `"ReviewReader"` or `"crate::traits::ReviewReader"`).
- **Canonical Name**: To get the reliable, fully qualified name, use the `id` from the `Path` to look up the item in the `Crate.paths` map. The `ItemSummary` in `paths` contains a `path: Vec<String>` field (e.g., `["sotohe_core", "domain", "ReviewReader"]`).

### 3. Handling Generics and Complex Impls

- **Generic Trait Impls**: Represented via `Impl.generics` (for the `impl<T>` part) and `Path.args` / `Type.args` (for the `Trait<T>` and `Struct<T>` parts).
- **Blanket Impls**: Detected when `blanket_impl` is `Some(Type)`. For `impl<T: Display> Foo for T`, the `for_` field will be `Type::Generic("T")` and `blanket_impl` will also reflect this type.
- **Auto-traits/Synthetic**: Identified by `is_synthetic: true`. These should be ignored for adapter validation.

### 4. Compatibility and Stability (0.50 to 0.57)

There are significant breaking changes between these versions:
- **v0.54**: `Item.attrs` changed from `Vec<String>` to `Vec<Attribute>` enum.
- **v0.57**: `Path.name` was renamed to **`Path.path`**.
- **v0.57**: Line numbers in `Span` are now guaranteed to be 1-indexed.
- **v0.57**: `ExternalCrate` gained a `path` field for better dependency resolution.

### 5. Verified Rust Snippet (rustdoc-types 0.57)

```rust
use rustdoc_types::{Crate, Item, ItemEnum, Type};

pub fn find_adapter_impls(krate: &Crate) {
    for item in krate.index.values() {
        if let ItemEnum::Impl(ref imp) = item.inner {
            // Filter: Skip synthetic (auto-traits), negative, and blanket impls
            if imp.is_synthetic || imp.is_negative || imp.blanket_impl.is_some() {
                continue;
            }

            if let Some(ref trait_path) = imp.trait_ {
                // Verified field name: trait_path.path (not .name)
                let trait_name = &trait_path.path;
                
                // Extract implementing type name
                if let Type::ResolvedPath(ref p) = imp.for_ {
                    let struct_name = &p.path;
                    println!("Found: impl {} for {}", trait_name, struct_name);
                }
            }
        }
    }
}
```

### 6. Recommendations for TDDD Infrastructure

- **Approach**: **Manual parsing via `rustdoc-types`** is the most maintainable path. High-level abstractions for rustdoc JSON are often out-of-date or don't support the latest nightly changes. A simple visitor pattern over `Crate.index` is sufficient for 1.3 MB of JSON.
- **Filtering**:
  - **Skip `is_synthetic`**: Excludes `Send`, `Sync`, `Unpin`, etc.
  - **Skip `is_negative`**: Excludes `impl !Trait`.
  - **Skip `blanket_impl.is_some()`**: For adapter validation, you only want concrete implementations (e.g., `FsReviewStore`), not generic transformations.
- **Resolution Strategy**: Always check `Crate.index` first (for local items), then fall back to `Crate.paths` (for external items) using the `Id`.
