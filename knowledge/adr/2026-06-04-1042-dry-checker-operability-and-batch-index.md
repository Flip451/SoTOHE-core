# dry-checker(sotp dry)の運用修正 — Codex/ChatGPT アカウント対応・出力スキーマ厳格化・インデックス成果物の除外・insert と埋め込みの一括化・インデックスの永続化

## Context

DRY 違反検出(`sotp dry write` / `dry check-approved` / `dry fix-local`)を実際の commit ゲート上で動かしたところ、検出機能(`2026-05-29-1118-semantic-dup-detection-discoverability-gate.md` の母体機能)が Codex/ChatGPT アカウント環境で通しで動かない不備が複数判明した。いずれも「検出ロジックの設計」ではなく「運用上の成立条件」に関わる:

- `dry write` のエージェントモデル既定値が literal `"codex"` で、ChatGPT アカウントの Codex CLI ではモデル名として無効(400 `invalid_request_error`)。候補ペアの分類が一件も走らない。
- 分類エージェントの構造化出力スキーマ(`response_format`)が OpenAI strict モード非準拠で 400(`invalid_json_schema`)。`refactor_proposal` が `properties` にあるのに `required` から漏れている。
- 検出に使う LanceDB セマンティックインデックスのローカル成果物が `.gitignore` 未登録で、ステージング操作で commit に混入しうる。
- インデックス構築がフラグメント単位の逐次 insert(1 行 = 1 トランザクション)で、ワークスペース規模のコーパス(約 400 ソースファイル)に対し数千トランザクションを生み、インデックスディレクトリが GB 級に膨張、1 回の実行が極端に遅くなる。

これらは検出の正しさ(類似判定・判定モデルの良否)とは独立した運用品質の問題であり、まとめて解消する。D4 の挿入一括化後も、埋め込みがフラグメント単位の CPU 推論のまま残り、コーパス構築の主たる遅延要因であることが判明した。insert(D4)・embedding(D5)の一括化後も、毎回コーパス全体を再埋め込みするため、反復実行(DFP ループ・commit ゲート)が実用的時間で完了しないことが判明した。

## Decision

### D1: dry write のエージェントモデルを agent-profiles から解決する

`sotp dry write` のエージェントモデルは、`--model` を明示指定しない場合、`agent-profiles.json` の `dry-checker` capability から解決する(`AgentProfiles::resolve_execution(capability_name, Final)`)。既に同等の解決を行っている `dry fix-local` と挙動を揃える。CLI の `--model` 既定をリテラル `"codex"` にしてそのままアダプタへ渡す従来動作を廃止し、`--model` は省略可能(明示値は上書きとして尊重)とする。

理由: リテラル `"codex"` は環境によって無効なモデル名であり、運用者が毎回モデルを明示しないと動かない。capability 解決にすれば、プロバイダ/モデルの設定一元化(agent-profiles)に従い、無人実行(dfl ループ・ゲート内部呼び出し)でも成立する。

### D2: 分類エージェントの出力スキーマを OpenAI strict 準拠にする

DRY 判定エージェントの構造化出力スキーマは、`required` 配列に `properties` の全キーを列挙する。`refactor_proposal` は違反でない場合に値を持たないため、型は nullable(`["string", "null"]`)を維持したうえで `required` に含める。スキーマの構造的不変条件(`required` ⊇ `properties` のキー集合)を回帰テストで固定する。

理由: OpenAI strict 構造化出力は `required` が `properties` の全キーを含むことを要求する。欠落があると API が 400 を返し、候補ペアの分類が成立しない。nullable を保てばモデルは非違反時に `null` を返せ、受け側の任意フィールド処理と整合する。

### D3: セマンティックインデックスのローカル成果物を gitignore する

DRY 検出が生成するローカル成果物を `.gitignore` に登録する。対象は次の通り:

- `.semantic_index/` — 永続インデックス本体(LanceDB データディレクトリ)
- `.semantic_index.*` — インデックスと同じ階層に生成される sidecar ファイル群。D6 が生成するキャッシュマーカー(`.semantic_index.sotp-cache`)・ファイルシステムロック(`.semantic_index.lock`)、D7 が生成するファイル単位マニフェスト(`.semantic_index.manifest`)がこれに含まれる。単一のグロブ `.semantic_index*` でディレクトリと全 sidecar を覆う。
- `sotp-dry-index-*/` — 一時インデックスディレクトリ

`.fastembed_cache/`(モデルキャッシュ)と同様、これらはすべて再生成可能なローカル成果物であり、バージョン管理の対象外とする。

理由: 永続インデックスは大量のトランザクションファイルを含み、sidecar 群(マーカー / ロック / マニフェスト)はインデックスの再利用・増分更新に用いる副産物で、いずれも実行環境ごとに生成される。未 ignore のままだと一括ステージングで commit に混入する。再生成可能でリポジトリに含める意味がない。

### D4: セマンティックインデックスへの insert を一括化する

コーパスのインデックス構築は、フラグメントを 1 件ずつ insert するのではなく、全フラグメントを 1 回(単一の RecordBatch + 単一の追加操作)で投入する。このため意味インデックスのポートに一括投入のオペレーションを追加し(`SemanticIndexPort::insert_batch`)、コーパス構築を行う利用側(DRY 書き込み・承認確認の各インタラクタ)が一括投入を呼ぶ。単件 insert は別の利用者が使うため残す。

理由: 1 フラグメント = 1 トランザクションだと、ワークスペース規模のコーパスで数千トランザクションが発生し、インデックスが肥大化して実行が運用に堪えない速度になる。一括投入はトランザクション数を定数に抑え、構築を大幅に高速化する。

### D5: コーパス埋め込みを一括化する

DRY 検出のコーパス構築における埋め込みは、フラグメントを 1 件ずつ embed するのではなく、埋め込みライブラリ(fastembed)の一括 API を使って全フラグメントを 1 回で embed する。このため意味インデックスの埋め込みポートに一括埋め込みのオペレーションを追加し(`EmbeddingPort::embed_batch`)、コーパス構築を行う共有ヘルパーが一括埋め込みを呼ぶ。単件 embed は他の利用者がいれば残す。

理由: D4 の挿入一括化でトランザクション集中は解消したが、埋め込みが依然フラグメント単位の CPU 推論のままで、コーパス構築の主たる遅延要因として残っている。fastembed は本来一括推論に対応しており、一括化によって構築を大幅に短縮できる。

### D6: セマンティックインデックスを永続化しコーパス不変時は再利用する

現状、`sotp dry write` / `dry check-approved` は呼び出しごとにフレッシュな一時インデックス(`sotp-dry-index-*`)を作り、コーパス全体を毎回ゼロから埋め込む(過去フラグメントの stale 蓄積を避けるための設計)。コーパス埋め込み(数千フラグメントの CPU 推論)が支配的コストとして毎回発生する。

決定: セマンティックインデックスを `--db-path`(既定 `.semantic_index`、D3 で gitignore 済み)に**永続化**し、**コーパスのフィンガープリント**(全コーパスフラグメントの (path, 内容ハッシュ) 集合 + 埋め込みモデル識別子から導出する単一ハッシュ)をインデックスと併せて記録する。実行時:

- 永続インデックスが存在し、記録済みフィンガープリントが現在のコーパスのものと一致 → インデックスを再構築せず再利用(埋め込み + 投入をスキップ)。
- 不一致(または不在)→ 全再構築(コーパスを埋め込み直し insert_batch し、新しいフィンガープリントを記録)。

これにより、コーパスが不変な場合(同一 run 内の write→check-approved、同一コードに対する連続実行)の再埋め込みを回避しつつ、コーパス変化時は全再構築するため stale フラグメントは混入しない。

永続インデックスの安全機構として、`sotp dry` が管理するキャッシュであることを示すマーカー(`.semantic_index.sotp-cache`)を作成・確認し、永続インデックス操作中はファイルシステムロック(`.semantic_index.lock`)で排他制御する。既存のインデックスパスがマーカーを持たない場合は管理外ディレクトリとして扱い、削除や上書きを行わず fail-closed する。

これはコーパス構築の共有ヘルパー + インデックスアダプタ(infrastructure)の変更を伴う。型変更(新 port メソッド等)が必要なら、本 ADR を spec が参照し型カタログに宣言して grounding する(設計を経ずに型を足さない)。具体的な実装位置・フィンガープリント計算・port シグネチャは track 実装で確定する。

理由: insert(D4)と embedding(D5)の一括化で 1 回あたりの構築は改善するが、毎回コーパス全体を再埋め込みする限り、commit ゲート(check-approved)や DFP ループは数千フラグメントの ONNX 推論コストを繰り返し負担し続け、実用的時間で完了しない。フィンガープリント・キーの再利用は、不変コーパスに対する反復コストを排除する。

### D7: セマンティックインデックスをファイル単位の内容ハッシュで増分維持する

D6 ではインデックスを永続化し、コーパス全体の単一フィンガープリント(全 (path, 内容ハッシュ) 集合 + 埋め込みモデル識別子の SHA-256)が一致すれば再利用、不一致なら全再構築する。しかし 1 ファイルでも変わるとフィンガープリントが不一致になり、コーパス全体(数千フラグメント)を毎回ゼロから再埋め込みする。実運用では各実行で変わるのは少数ファイルのため、この全再構築が支配的コストとして残る。

決定: 単一フィンガープリントに代えて、インデックスに含まれる各ソースファイルの内容ハッシュを記録したマニフェスト(`{path → 内容ハッシュ}` と埋め込みモデル識別子)を永続化する。実行時、現在のワーキングツリーの `{path → 内容ハッシュ}` をマニフェストと差分比較し、変化分だけを更新する:

- 内容ハッシュが変わったファイル / 新規ファイル → そのファイルの既存フラグメントをインデックスから削除(`source_path` 単位)し、現在のフラグメントを抽出・埋め込み・挿入する。
- 消えたファイル → そのフラグメントを削除する。
- 不変のファイル → 何もしない(再埋め込みしない)。

差分が空(何も変わっていない)の場合は削除も挿入も発生せず、D6 の「コーパス不変時は再利用」を包含する。埋め込みモデル識別子が変わった場合は埋め込み空間が変わるため全ファイルを dirty 扱いとして全再構築する。

この増分維持には意味インデックスのポートに「`source_path` を指定してそのファイルの全フラグメントを削除する」オペレーションが要る。1 ファイルは複数フラグメント(関数等)が同一 `source_path` を共有するため、「`source_path` で削除して入れ直す」が 1 ファイル再索引の正しい単位になる。

D6 との関係: 本決定は D6 の「単一フィンガープリント + 不一致時の全再構築」という再利用機構を「ファイル単位マニフェスト + 増分更新」に置き換える。D6 のインデックス永続化(`--db-path` への保存)とキャッシュ安全機構(マーカー / ロック)はそのまま維持する。

鍵を git のコミットハッシュではなくワーキングツリーの内容ハッシュにする理由: DRY のコーパスはコミット前・未追跡を含むワーキングツリーの現在内容から抽出される。コミットハッシュは未コミット状態を表せず、ブランチ操作でハッシュが到達不能・巨大差分になりうる。内容ハッシュはコーパス抽出と同じ真実源で動くため未コミット/未追跡も正しく扱え、git にも依存しない。内容のハッシュ計算は埋め込みに比べ桁違いに安く、支配コストである再埋め込みを変化分だけに削減できる。

理由: 反復実行(DFP ループ・commit ゲート)では各回の変更は少数ファイルに留まるのが通常で、全再構築は無駄が大きい。増分維持により、初回構築後は変化したファイルの再埋め込みだけで済み、反復実行が実用的時間で完了する。

## Rejected Alternatives

### A. モデル既定値を有効なモデル名にハードコードする(D1 の代替)

`--model` 既定を `"codex"` から特定の有効モデル名に差し替える案。モデル名が CLI と agent-profiles の二箇所に分散し、プロバイダ/モデル変更時に乖離する。設定の一元化に反するため却下。capability 解決に寄せる。

### B. 埋め込みを per-fragment のまま許容する(D5 の代替)

フラグメント単位の embed を維持し、構築の遅さを受け入れる案。D4 の挿入一括化後もコーパス構築が遅く、commit ゲートや DFP ループの実用性が回復しない。fastembed は一括推論を提供しており、利用しない理由がない。却下。

### C. insert を逐次のまま許容する(D4 の代替)

per-fragment insert を維持し遅さを受け入れる案。ワークスペース規模で 1 実行が極端に遅く、commit ゲートや DFP ループの運用に堪えない。却下。

### D. インデックス構築の対象を狭めることで遅さに対処する(D4 の代替)

コーパス走査の対象ディレクトリを絞ることで件数を減らす案。走査側は既に `target` および隠しディレクトリ(`.cache` 等)を除外しており、対象はワークスペースソースに限定済みで、走査範囲の問題ではない。遅さの主因は投入のトランザクション粒度であり、一括化(D4)が正しい対処。却下。

### E. フラグメント単位の差分更新(変更フラグメントのみ再埋め込み)(D6 の代替)

コーパスの変更フラグメントのみを特定して再埋め込みし、インデックスを部分的に更新する案。正確な差分追跡・古いフラグメントの部分無効化が必要で実装が複雑になる。まずフィンガープリント全体一致による全体再利用/全体再構築で反復コストを解消できれば十分であり、差分更新は将来の最適化として検討の余地を残しつつ、現時点では却下。

### F. フレッシュ一時インデックスのまま維持する(D6 の代替)

D6 を採用せず、呼び出しごとに一時インデックスを作り続ける案。D4・D5 の一括化後も、毎回コーパス全体を再埋め込みするコストが繰り返し発生し、反復実行(DFP ループ・commit ゲート)の実用的完了が得られない。却下。

### G. git コミットハッシュ + git diff による増分(D7 の代替)

サイドカーにインデックス構築時の git コミットハッシュを記録し、次回 `git diff <記録ハッシュ>` で変化ファイルを求めて増分更新する案。却下理由: DRY のコーパスはワーキングツリー(未コミット・未追跡を含む)から抽出されるため、コミットハッシュでは構築時・実行時の未コミット状態を表現できず取りこぼす。ブランチ切替・rebase・gc で記録ハッシュが到達不能・巨大差分になり、フォールバック全再構築が頻発する。git への結合も増える。ファイル単位の内容ハッシュ方式が同等の速度向上をより堅牢に(git 非依存・ワーキングツリー整合)達成する。

## Consequences

### Positive

- `sotp dry` 一式が Codex/ChatGPT アカウント環境で通しで成立する(モデル解決・スキーマ整合)。
- インデックス成果物が commit を汚さない。
- インデックス構築が大幅に高速化し、commit ゲート・DFP ループの実用性が回復する。
- コーパス埋め込みが一括化され構築が大幅に高速化する。
- 不変コーパスに対する反復実行で再埋め込みを回避し、DFP ループ・commit ゲートが実用的時間で完了する。

### Negative

- ポートに一括投入オペレーションが増え、ポートを実装/モックする箇所が追従を要する。
- 単件 insert と一括 insert の二経路が併存する(単件は別利用者が使うため残置)。
- 埋め込みポートに一括埋め込みオペレーションが増え、ポートを実装/モックする箇所が追従を要する。
- 単件 embed と一括 embed の二経路が併存しうる(単件は他の利用者がいれば残置)。
- 永続インデックスのマニフェスト管理(ファイル内容ハッシュ・埋め込みモデル識別子・差分判定)とキャッシュ安全機構(マーカー / ロック)が増える。差分判定やキャッシュ所有権判定を誤ると stale 再利用や管理外ディレクトリ破壊のリスクがあるため、実装では内容ハッシュに基づく厳密な dirty/deleted 判定、モデル変更時の全再構築、マーカー不在時の fail-closed に倒す。

### Neutral

- DRY 検出の判定基準(類似閾値・判定モデルの良否)は不変。運用成立条件のみを変える。

## Reassess When

- エージェントのモデル/プロバイダ解決機構(agent-profiles の構造)が変わった場合。
- 採用するベクトルインデックス実装(LanceDB)や投入 API が変わり、トランザクション粒度の前提が変化した場合。
- 採用する埋め込みライブラリ(fastembed)や一括 API の仕様が変わった場合。
- 構造化出力の API 仕様(strict モードの要件)が変わった場合。
- コーパス規模/埋め込みコストが変わり全体再構築前提が崩れた場合(差分更新の検討)。
