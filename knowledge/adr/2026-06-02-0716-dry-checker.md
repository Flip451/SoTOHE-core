# semantic-dup を活用した DRY 違反の自動検出 capability

## Context

semantic-dup 機能（`sotp find-similar` / `dup-check` / `dup-index build` / `dup-index measure-quality`）は、実装者がコミット前に類似する既存実装を見つける「実装時の発見支援」と、差分フラグメントを意味DBに照会する soft gate を提供する。ただし「実装が既存コードを意味的に重複（DRY 違反）していないか」をレビュー時に検証する手段は、現状では人手に頼っている。

これらの既存コマンドと、類似候補を本物の DRY 違反かどうか判定するエージェント層を組み合わせれば、実装の DRY 性を自動で検証する capability を設けられる。capability システム（`agent-profiles.json` による provider 振り分け）とスコープ単位の並列レビュー基盤も既にあるため、技術的な土台は揃っている。

一方で、設計上の制約が 3 つある。

- **実行タイミング（いつ走らせるか）**: DRY 検証とレビューを同時並列で走らせると、両者が指摘する修正範囲が重複し、「どちらの指摘に従うべきか」「自動修正が競合する」といった調停が厄介になる。したがって DRY 検証とレビューを同時並列で走らせる必要はなく、むしろ走らせるべきではない（実行順序を分ける）。これは timing の問題。
- **結合度（どう作るか）**: DRY ゲートとレビューゲートは本来独立した別機能である。既存のレビュー基盤の設計は参考にしてよいが、流用しすぎて両ゲートが密結合になると、片方の変更が他方へ波及し、独立して進化させられなくなる。両者は疎結合に保つ必要がある（これは architecture の問題で、上の timing とは別軸）。
- **モデル配布の前提**: semantic-dup の embedding は外部ネットワーク通信なしのオフライン専用で、実行前に embedding モデル（Jina v2 base code, 約 550MB）をローカルキャッシュへ事前配置しておく必要がある。DRY 検証を CI などネットワーク隔離環境で動かす場合、その環境へモデルをどう届けるかが前提のブロッカーになる（この配布方法は D6 で決定する）。

## Decision

### D1: DRY ゲートはレビューゲートと疎結合な別ゲート/別コマンドにする（結合度）

これは「どう作るか（architecture）」の決定であり、「いつ走らせるか（timing）」は D2 で扱う。DRY 検証はレビューコマンドに組み込まず、独立した専用コマンド（例: `sotp` の DRY 検証サブコマンド / `/track:dry-check` 相当）として起動する。既存のレビュー基盤の設計は参考にしてよいが、流用しすぎて両ゲートが密結合にならないようにし、インターフェースを疎結合に保って相互に依存させない（片方の変更が他方へ波及しないようにする）。DRY 検証用の capability も reviewer とは別に設ける（D3）。

### D2: DRY fix phase（DFP）→ review fix phase（RFP）の 2 フェーズ。RFP が DRY を壊したら DFP へ戻る（実行タイミング）

これは「いつ走らせるか（timing）」の決定であり、D1 の結合度とは別軸。DRY とレビューを同時並列で走らせると修正範囲が重複し調停が厄介になるため、時系列で分離した 2 フェーズにする。

- **DFP（DRY fix phase）**: DRY check + fix を DRY gate 通過まで回す（単一・全コードベース、D13）。
- **RFP（review fix phase）**: DFP 完了後に移行。per-scope 並列で review + fix を全 scope `zero_findings` まで回す（D13）。
- **back-edge（RFP → DFP）**: RFP の修正はコードを変えるため新たな重複を生みうる。RFP 中に DRY check が違反を捉えたら、rfl は即座に RFP を停止して DFP へ移行する（rfl は DRY を修正しない。DRY の修正は dfl が担う。D12）。DFP の修正でコードが変われば該当 review scope の hash が変わり `required` に戻る（既存の per-group hash ベース再 review 機構をそのまま使う）。
- **終端**: DRY gate と全 review scope が同時に green になった fixpoint でのみ commit 可（どちらか一方だけ green では commit しない）。

### D3: dry-checker capability + 専用コマンドを新設する

`agent-profiles.json` に `dry-checker` capability（判定役）を新設し（provider は Claude または Codex）、専用コマンドから起動する。reviewer / researcher とは別 capability とし、レビュー系の capability に DRY dimension を相乗りさせない（密結合回避）。修正役は DRY と review で分離する（dfl: dry-fix-lead / rfl: review-fix-lead、D12）。

### D4: DRY チェックのたびに最新コードから単一索引を構築し、追加/変更 hunk に重なるフラグメントを照会する

- **索引**: DRY チェックのたびに、最新（変更適用後）コードをソースとして単一索引を構築する。
- **照会**: git diff の**追加/変更された hunk に重なるフラグメント**のみをこの索引に照会し、閾値超の候補を上位 k 件取得して D5 の agent に渡す。照会対象を「この変更が実際に追加/変更したコード」に限ることで、DRY ゲートを当該変更にスコープする。
- **責務分担**: 検索層（D4）は recall を担い、precision のうち「この変更が導入/悪化させていない既存重複の除外」は hunk スコープで構造的に担保する（後述「粒度」参照）。自己/修正-自己・許容類似の除外は D5 の agent が担う。
- **粒度（hunk スコープ）**: フラグメントは意味単位（関数/ブロック）で切り出す。照会対象の判定は hunk 粒度で行い、**フラグメントのいずれかの行が追加/変更 hunk に含まれる場合にそのフラグメント全体を照会対象とする**。まったく hunk に重ならないフラグメントは照会しない。これにより、触れただけで内容を変えていない既存フラグメント（そこに存在していた既存重複を含む）は構造的に照会対象外となり、CN-04（未変更コード同士の既存重複でブロックしない）を LLM 判断に依らず決定的に保証する。
- **diff 取得**: diff の取得は、ファイルパスだけでなく**追加/変更された行範囲（hunk の行範囲）も含む情報**を返す。これにより上記の hunk スコープ判定が実現できる。
- **diff base の解決**: 「追加/変更された hunk」を求める際の diff 基点は、review と同一の動作で解決する。具体的には、各 track の `.commit_hash`（その track の直近コミットの SHA を記録したローカルファイル）を読み、それが現在の HEAD の祖先であればそれを diff 基点とし、祖先でない・不在なら `main` の tip にフォールバックする（fail-closed: 基点が無効なら diff 範囲が広がる側に倒す）。dry-check は review と同じ `.commit_hash` ファイルおよび低レベルの git 操作を参照するが、この解決は dry-check 専用の独立したインターフェース/アダプタを通じて実装する。review ゲートのインターフェース（`DiffGetter` / commit-hash reader 等）は共有・依存しない（D1 の疎結合原則: 両ゲートは `.commit_hash` という共通の artifacts と git の仕組みを共有するが、ゲートの interface は共有しない）。明示的な基点指定（`--base-commit` 等）は任意の上書き手段とし、既定はこの解決結果とする。

### D5: raw similarity の DRY 違反判定は dry-checker agent が行う（precision を担う）

閾値超過の類似候補をそのまま違反とはせず、`dry-checker` agent が候補ペアと実コードを読んで判定する。検索層（D4）が recall 重視で偽陽性を含む候補を渡すことを前提に、agent が precision を担保する。具体的には次を **却下**する:

- (i) **自己マッチ / 修正-自己**: 照会フラグメント自身、またはその変更前バージョン（同一 path・ほぼ同一内容など「編集しただけの同一コード」で、新しい重複ではないもの）。
- (ii) **許容される類似**: 層をまたぐ意図的なミラー（domain VO と infra DTO 等）、テスト、ボイラープレートなど。
- (iii) **スコープ外**: 未変更コード同士の既存重複は D4 の hunk スコープにより構造的に照会対象外となるため、原則 agent には渡らない。万一 hunk に重なる形で既存重複の残余が現れた場合には agent も却下する。agent の主務は (i)(ii) の precision 判定であり、(iii) は D4 の構造的排除が主に担う。

残った「この変更が導入/悪化させた本物の DRY 違反」のみをリファクタ提案付きで findings 化する。これが capability の付加価値であり、機械的な閾値判定だけでは誤検出が多くなる点を避ける。

### D6: embedding モデルは named volume / cache に初回取得し再利用して配布する

embedding モデルは、ネットワークに接続できる環境（開発者環境やセットアップ・事前取得ステップ等）で一度だけオンライン取得して named volume / cache に保存し、以降はその named volume をオフラインで再利用する方式で配布する。モデルをツールイメージへ焼き込む方式（再現性は高いがイメージが肥大化する）は採らない。ネットワーク隔離 CI は、事前に取得済みの named volume をそのまま再利用して実行するため、CI 実行中のオンライン通信は不要。named volume にモデルを届ける具体的な手順（どのステップが volume を作成・共有するか）は実装時の詳細とする。

### D7: DRY ゲートは全 above-threshold ペアの verdict が確定するまで進めない blocking gate にする

DRY 検証は、現在の変更フラグメントの全 above-threshold ペアの agent verdict が `not-a-violation`（偽陽性として却下）または `accepted`（真の重複だが許容として承認）のいずれかになるまで先に進めない blocking gate とする。未解決の重複（`violation` のままのペア）が残ったまま次工程（レビュー等）へ進めない。これは D2「review 前に解消」と整合する。

ここでいう「明示的に通す」とは、許容できる重複を黙殺するのではなく、D5 の precision 判断に基づいて dry-checker agent が **`verdict: "accepted"`** として dry-check.json に記録する（D9）ことを指す。この acceptance 判断は agent が担う。別の人間/dfl 向け ack コマンドや flat な ack-file は設けない（D9 に統合）。

genuine な `violation`（agent が違反と判定したもの）には ad-hoc な人間による許容の抜け道は無い。コードを変えて重複関係が変われば、D8 のフラグメント識別子（path, content_hash）の変化により該当ペアのキーが変わり未記録（未検証）扱いに戻り、agent が改めて判定する。これが「先に進む前に解消する」という blocking gate の性質。

### D8: 変更フラグメント↔候補ペアの検証状態をキャッシュし、検証済みペアは再サジェストしない

理由: 閾値超の候補数が上位 k を超えると、類似度の高い偽陽性（修正-自己や許容類似は真の違反より類似度が高くなりやすい）が上位 k を占め、真の DRY 違反が k+1 件目以降に埋もれて agent に渡らないことがある。

- (変更フラグメント, 候補フラグメント) ペアごとに agent の verdict をキャッシュする。
- **フラグメントの識別子**: 各フラグメントの識別子は **`(リポジトリ相対パス, content_hash)`** のペアとする。`content_hash` はそのフラグメントの内容（関数/ブロック単位）の SHA-256 ハッシュである。
- **ペアの識別子（`DryCheckPairKey`）**: 2 つのフラグメント識別子 `(path, content_hash)` を `(path, content_hash)` の辞書順でソートした **順序不変ペア `(low, high)`**（`low ≤ high`）とする。これによりファイルをまたぐ完全コピー（別パスで内容が同一のフラグメント）は `low.path ≠ high.path` のため別ペアとして独立したキーを持ち、一方の verdict が他方を隠さない。同一ファイル内の内容が異なる 2 フラグメント（`low.hash ≠ high.hash`）も別ペアとして扱われ、同一ファイル内の近似重複を判定できる。
- **内容ハッシュは識別子の一部**: 識別子に `content_hash` が含まれるため、フラグメントの内容が変わると `content_hash` が変わり → 識別子が変わり → 新しいペアキーになり → 過去のレコードと一致しなくなる → 未記録として再検証される。これにより「内容変更 → 再検証」が識別子マッチングによって暗黙に実現される（content_hash を識別子とは別の無効化ステップで比較する必要はない）。
- **自己マッチの除外**: 2 つのフラグメント識別子が等しい（`low == high`、すなわち `path` も `content_hash` も一致する）場合を自己マッチとし、agent に渡す前に除外して verdict を記録しない。これはフラグメント自身 vs 自身、および同一ファイル内の byte-identical なフラグメント同士を除外する。後者（同一ファイル内 byte-identical）はキーが衝突するため記録しないという**受容する狭い gap**である（同一ファイル内の近似重複は hash が異なるため除外されない）。
- 候補のサジェストでは検証済みペア（識別子が一致するレコードがある = 同一ペアキーのレコードが存在する）を除外する。
- DRY チェックは、未検証かつ閾値超の候補が無くなるまで「照会 → 自己マッチ除外 → agent 検証 → verdict をキャッシュ → 検証済みを除外して再照会」を繰り返す。
- 検証は冪等・増分的になり、再実行ではペアキーが一致するレコードがない（内容変更で新識別子になったペアを含む）未検証のペアだけが対象になる。D7 で「許容として通す」とされる `accepted` verdict はこのキャッシュの verdict の一種として扱われる（D9）。

### D9: DRY 検証状態を review.json に相当する per-track 永続ファイルに記録する

D8 の検証状態キャッシュを、review.json に相当する per-track の永続ファイル（例: `track/items/<id>/dry-check.json`）として設計する。ファイルは schema バージョン付きエンベロープ形式とし、履歴はエンベロープ内の `records` 配列として保持する。インフラ層のコーデックはこの D9 が定めるスキーマを実装する。

**リファクタ提案の永続化（enum-first 設計）**: violation レコードはリファクタ提案を永続フィールドとして持つ。提案は `violation` という判定結果と切り離せない情報であるため、nullable/optional フィールドではなく `verdict` 自体を enum-first で設計し、`violation` の場合のみ `refactor_proposal` を構造的に持たせる。これにより「violation なのに提案がない」「violation でないのに提案がある」という不正な状態が構造的に作れなくなる（`not-a-violation` / `accepted` は提案フィールドを持たない）。serde タグの具体的な表現形式はインフラ層のコーデック実装の詳細とする。

**ファイル構造（dry-check.json のスキーマ定義）**:

```jsonc
// illustrative, non-canonical
{
  "schema_version": 1,
  "records": [
    {
      "low_path": "<フラグメント識別子 (path, content_hash) を辞書順ソートしたとき low 側のパス>",
      "low_hash": "<low 側フラグメントの内容ハッシュ（SHA-256）>",
      "high_path": "<フラグメント識別子 (path, content_hash) を辞書順ソートしたとき high 側のパス>",
      "high_hash": "<high 側フラグメントの内容ハッシュ（SHA-256）>",
      "changed_path": "<diff フラグメント側のパス（表示専用・ペア識別子でも無効化比較でも使わない）>",
      "verdict": { "violation": { "refactor_proposal": "<リファクタ提案テキスト>" } },
      "similarity_score": 0.92,
      "threshold": 0.80,
      "base_commit": "<diff の基点となった git コミットハッシュ>",
      "rationale": "<agent の判定根拠（なぜ違反と判定したかの理由）>",
      "recorded_at": "2026-06-02T07:16:00Z"
    },
    {
      "low_path": "<フラグメント識別子 (path, content_hash) を辞書順ソートしたとき low 側のパス>",
      "low_hash": "<low 側フラグメントの内容ハッシュ（SHA-256）>",
      "high_path": "<フラグメント識別子 (path, content_hash) を辞書順ソートしたとき high 側のパス>",
      "high_hash": "<high 側フラグメントの内容ハッシュ（SHA-256）>",
      "changed_path": "<diff フラグメント側のパス（表示専用）>",
      "verdict": "not-a-violation",
      "similarity_score": 0.85,
      "threshold": 0.80,
      "base_commit": "<diff の基点となった git コミットハッシュ>",
      "rationale": "<却下理由（許容類似など）>",
      "recorded_at": "2026-06-02T07:16:00Z"
    }
  ]
}
```

各フィールドの意味:

- **`schema_version`**: エンベロープのスキーマバージョン（現在は `1`）。将来の後方互換管理用。
- **`records`**: 検証済みペアのレコード配列。追記型の履歴として扱う。
- **`low_path` / `low_hash` / `high_path` / `high_hash`**: この **4 フィールドが合わさってペア識別子（`DryCheckPairKey`）を構成する**（D8）。各フラグメントの識別子は `(リポジトリ相対パス, content_hash)` であり、2 つの識別子を `(path, content_hash)` の辞書順でソートして `low`（小さい方）と `high`（大きい方）に割り当てる。同一ペアは `(A, B)` で来ても `(B, A)` で来ても同じ `low`/`high` に落ち着く。パスが異なれば内容ハッシュが同一でも別ペアとなる（別ファイルの完全コピーが 1 件に折りたたまれない）。同一ファイル内で内容が異なる 2 フラグメントは `content_hash` が異なるため別ペアとなり判定できる。自己マッチ（`low_path == high_path` かつ `low_hash == high_hash`）は記録しない（D8）。フラグメントの内容が変わると `content_hash` が変わり → ペアキーが変わり → 過去レコードと一致しなくなる → 未記録として再検証される（無効化は識別子マッチングに内包されており、別途ハッシュ比較ステップは不要）。
- **`changed_path`**: diff フラグメントとして実際に照会された側のパス（`low_path` または `high_path` のどちらか）。**表示専用フィールド**であり、ペアの識別子にも無効化の判定にも使わない。`sotp dry results` での「どちらが変更フラグメントだったか」の表示に利用する。
- **`verdict`**: 判定結果。enum-first で設計する（上記参照）。`not-a-violation`（偽陽性として却下）/ `accepted`（真の重複だが許容として agent が承認、D7 の blocking gate 通過条件の一方）はそのまま文字列の bare 値。`violation`（違反）の場合のみ `refactor_proposal` フィールドを持つオブジェクト型（`{ "violation": { "refactor_proposal": "..." } }` 相当）とし、提案を永続フィールドとして保持する（`sotp dry results` での表示に利用）。`not-a-violation` / `accepted` はリファクタ提案を持たない。
- **`similarity_score`**: チェック時点での取得類似度スコア（0.0〜1.0）。
- **`threshold`**: チェック時点で有効だった類似度閾値（0.0〜1.0）。
- **`base_commit`**: diff の計算基点となった git コミット（チェックの差分文脈）。
- **`rationale`**: agent の判定根拠テキスト（必須・非 null）。`violation` の場合はなぜ違反と判定したかの理由、`not-a-violation` の場合は却下理由（許容類似など）、`accepted` の場合は許容とした根拠を記す。リファクタ提案（violation の場合）は `verdict` 内に保持し、`rationale` はすべての verdict で共通の「判定根拠（理由）」のみを記す。agent は verdict を出す際に必ず判定根拠を生成するため、このフィールドは常に存在する。省略を許すと「根拠のないレコード」という不正な状態が発生するため、必須とする。
- **`recorded_at`**: 記録日時（ISO-8601 UTC、必須・非 null）。インフラ層の CLI 書き込みコマンドが追記時に必ずタイムスタンプを付与するため、このフィールドは常に存在する。

- **最新エントリが有効**: 同一ペア識別子 `(low_path, low_hash, high_path, high_hash)` に対して複数のレコードがある場合（同一内容での再実行）、配列の最後のエントリの verdict を現在の検証結果とする（追記型の履歴）。これにより D8 の「ペア識別子をキーに verdict をキャッシュ」という概念が維持される。
- **再検証（識別子マッチングによる暗黙の無効化）**: フラグメントの内容が変わると `content_hash` が変わり → ペア識別子が変わり → 過去レコードと一致しなくなる → 未記録として再検証される。別途「現在の内容ハッシュと記録ハッシュを比較する」無効化ステップは不要（識別子マッチングが無効化を兼ねる）。パスを移動した場合（内容同一・パス変更）も同様に新しい識別子で未記録となるため再検証される（許容しうるコスト）。
- **commit**: review.json と同様、track の検証記録として commit する。
- **acceptance の記録**: D7 で「許容として通す」とされる判断はこのファイルの `verdict: "accepted"` として記録する。別の flat な ack-file は設けない（D7 の blocking gate は `not-a-violation` または `accepted` のいずれかで通過する）。

### D10: DRY checker の操作を「検証&記録」「読み出し」「ゲート判定」の 3 経路に分離する

review システム（reviewer 実行 + review.json への記録 / `sotp review results` での読み出し / `track-check-approved` でのゲート判定）の分離に倣い、3 操作を別経路に分ける。dry-check.json（D9）への直接読み書きを各所に散在させない。

1. **DRY 検証 & verdict 記録（write）**: 索引構築（D4）→ 候補照会 → agent 検証（D5）を実行し、ペア識別子と一致するレコードがない（フラグメント内容変更で新識別子になったペアを含む）未検証ペアのみ検証して（D8）verdict を dry-check.json に記録する。dry-check.json への書き込みはこの経路のみとする。
2. **記録の読み出し（read）**: 専用の読み出しコマンド（例: `sotp dry results`）で、記録済みの候補ペアとその verdict 分類の一覧を表示する。表示する verdict 分類（違反でない / 許容として承認 / 違反）はフラグで絞り込める。これは記録の情報確認 (informational listing) であり、gate（後述の bullet 3）の判定は行わない。dry-check.json を直接 read しない（review.json 直読み禁止と同じ理由: 権限プロンプト・改竄/自己参照の回避）。
3. **DRY ゲートのチェック（gate）**: 専用コマンド（例: `sotp dry check-approved`、track-check-approved 相当）で「現在の変更フラグメントの全 above-threshold ペアが検証済み かつ verdict ∈ {違反でない, 許容として承認}」を判定する。ペア識別子が一致するレコードがない未検証ペアや、未解決の verdict=違反が 1 つでもあれば gate は blocking とする（D7）。

### D11: verdict 記録は schema 制約付き agent 出力を CLI が書き込む（改竄防止・直接書き込み禁止）

review.json の autorecord と同様、dry-check.json への書き込みは「schema 制約付きの agent 出力を CLI コマンドが取り込んで追記する」経路に限定し、orchestrator や dfl/rfl（D12）が dry-check.json を直接 Edit/Write しない。

- dry-checker agent は構造化出力制約（Codex の `--output-schema` 等、provider 相当の機構）で verdict を schema 準拠の JSON として出力する。agent JSON が含むのは **verdict・判定根拠（rationale）・violation 時のリファクタ提案（refactor_proposal）のみ**。フラグメントの識別子（path / content_hash）およびペア識別子（DryCheckPairKey）は agent JSON に含めない。
- **フラグメント identity（ペア識別子 DryCheckPairKey = low/high にソートした (path, content_hash) ペア）は CLI/adapter が計算する**。具体的には、judge に渡した実際の CodeFragment の `source_path` と fragment content の SHA-256 から導出する。agent 出力には依存しない。これにより identity は source content 由来となり、agent が identity を偽造することを構造的に排除できる（D8 の identity 契約と整合し、D7/D8 の改竄耐性をさらに強化する）。
- CLI コマンド（D10 の write 経路）が、agent の schema 検証済み JSON（verdict/rationale/refactor_proposal）と、CLI が計算したペア識別子を合わせて dry-check.json に追記する。dry-check.json を書き込むのはこの CLI 経路のみ。
- orchestrator / dfl・rfl（D12）/ その他 agent は dry-check.json を直接編集しない。verdict は必ず「judge（dry-checker capability）の schema 出力 → CLI 書き込み」を経る（「違反なし」を手で書いてゲートを通すような改竄を構造的に防ぐ）。identity の計算も CLI が担うため、LLM 出力を identity の権威にしない二段構えで改竄耐性が成立する。

### D12: DRY と review の修正役は分離する（dry-fix-lead と review-fix-lead を一つにまとめない）

修正役を一つの fix-lead に統合すると、DFP と RFP が単一 agent の内部ループに溶け込み、phase 境界が消えて密結合になる。各 phase に単一責務の専用 fix-lead を置いて分離する（D1 の gate/judge/state 分離を fixer にも及ぼす）。

- **dry-fix-lead（dfl）**: DFP で DRY 違反のみを修正する（whole-codebase, D13）。
- **review-fix-lead（rfl）**: RFP で review 指摘のみを修正する（per-scope 並列, D13）。既存の rfl をそのまま使い、DRY 修正ロジックを混ぜない。
- **rfl は DRY を修正しない**: RFP 中に DRY check が違反を捉えたら、rfl は即座に RFP を停止して DFP へ移行する（D2 の back-edge）。DRY の修正は dfl が担う。phase 間の遷移は agent 内部のモード切替ではなく明示的なハンドオフとする。
- どちらの fix-lead も verdict（dry-check.json / review.json）を直接書かない（D11）。判定役（dry-checker / reviewer）と修正役の分離も維持する。

### D13: DRY は単一（全コードベース）スコープの dfl、review は per-scope 並列の rfl

review は scope（cli / domain / infrastructure / ...）ごとに分割され per-scope の rfl が並列実行される。一方 DRY 違反は scope/層をまたぐ（domain の片が infrastructure の片と重複しうる）ため partition できず、DRY は単一の全コードベーススコープで dfl が扱う（D4 の単一索引と整合）。

- **dfl（DFP）**: 全コードベースを 1 つのスコープとして単一セッションで駆動する。重複解消のリファクタは scope をまたいで編集してよい（修正の boundary = 全コードベース）。状態は per-track 単一の dry-check.json（D9）。
- **rfl（RFP）**: 従来どおり scope ごとに分割し、per-scope で並列実行する。各 rfl は自分の scope 内に編集を限定する。状態は review.json の per-scope エントリ。
- **順序**: D2 のとおり DFP（dfl, 単一）を先に gate 通過させてから RFP（rfl, per-scope 並列）に入る。RFP 中の DRY 違反は rfl が RFP を停止して DFP へ戻す（D2/D12）。
- DRY を review の 1 pseudo-scope に押し込まない（DRY は partition でなく全体かつ別 gate であり、D1 の疎結合に反するため）。

## Rejected Alternatives

### A. review gate への統合（密結合）

reviewer capability に DRY dimension を追加し、レビューと一体で運用する案。却下理由: DRY 検証の修正範囲がレビュアーの修正範囲と競合し、レビューゲートと DRY ゲートが密結合になる（D1 / D3 の方針に反する）。

### B. raw 閾値の機械判定のみ

agent 判定層を設けず、閾値超過をそのまま違反として findings 化する案。却下理由: 層をまたぐ意図的なミラー・テスト・ボイラープレートなどで誤検出が過多になり、ノイズで使い物にならなくなる（D5）。

### C. 索引を baseline（変更前コード）のみに限定する

意味DBの索引を baseline（変更前）コードのみで構築する案。却下理由: 変更セット内の新規コード同士の重複を取りこぼす（D4）。

### D. 構文的 DRY 検出（非 embedding）

AST / トークン一致など構文ベースの重複検出で embedding を避ける案。却下理由: 「同等ロジックの別表現」のような意味的な重複を見逃す。また semantic-dup の既存資産（意味DB・コマンド群）を活かせない。

## Consequences

### Positive

- review と独立した DRY 検証で、既存実装の再発明を構造的に抑止できる（実装時の発見支援を review 時へ拡張する）。
- semantic-dup の既存コマンド資産を再利用するため、新規実装が最小で済む。
- agent 判定層により誤検出を抑え、層をまたぐ意図的な類似を違反と誤認しない。
- hunk スコープ（D4）により、触れただけで内容を変えていない既存フラグメント（既存重複を含む）を LLM 判断に依らず構造的に照会対象外にできる（CN-04 を決定的に保証）。
- blocking gate により「許容する重複は agent が `accepted` として明示的に記録し、未解決の違反は先に進めない」を担保できる。
- 検証状態キャッシュ（D8）により、偽陽性に埋もれた真の違反の見逃しを防ぎ、再実行を冪等・増分的にできる。
- verdict 記録を schema 制約付き agent 出力 → CLI 書き込みに限定する（D11）ことで、orchestrator/dfl・rfl が「違反なし」を手書きしてゲートを通す改竄を構造的に防げる（review.json autorecord と同じ防御）。

### Negative

- モデル配布（ネットワーク隔離環境へモデルをどう届けるか）と、DRY チェックごとの索引構築（最新コードの embed）にコストがかかる。索引はキャッシュ/増分更新で最適化しうる（実装の最適化事項）。
- DRY 検証をレビュー前段に置くため、ワークフローに 1 ステップ増える（実装 → DRY 解消 → レビュー）。
- precision を agent に寄せた分（D4 は recall 重視）、agent は自己/修正-自己・許容類似といった却下対象も含めて多めの候補を実コードで読むため、コストとレイテンシが増える（機械的に前段フィルタする場合より agent の負荷は高い）。既存重複は hunk スコープで構造的に除外されるが（D4）、閾値設定で候補数を抑える調整が要る。
- 別 capability / コマンドを保守する負担（レビュー基盤と疎結合に保つための設計上の配慮が要る）。
- フラグメント識別子 `(path, content_hash)` のソート済みペアをキーに verdict を記録する永続状態（dry-check.json）を保守する必要がある（D8/D9）。識別子マッチングが無効化を兼ねるため別途ハッシュ比較ロジックは不要だが、同一ファイル内の byte-identical なフラグメント同士（自己マッチとして除外）は記録されないという狭い gap がある。

### Neutral

- 近似最近傍探索の非決定性は、固定スナップショットと閾値設定で再現性を確保する必要がある（semantic-dup が「CI の再現性を壊さない」ために採った方針を踏襲する）。

## Reassess When

- semantic-dup の embedding モデルや閾値設定が変わったとき。
- 誤検出率が高く、agent 判定層の運用が重いと判明したとき。
- レビュー基盤との疎結合が崩れ、密結合の兆候が出たとき。
- 索引構築・キャッシュコストが許容できなくなったとき。
- DFP↔RFP の back-edge ループが収束しない（振動する）ケースが観測されたとき。
